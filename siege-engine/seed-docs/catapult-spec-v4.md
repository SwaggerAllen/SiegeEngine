# Catapult — Specification (v4)

**Status:** in-progress prose draft. v4 supersedes v3's TOC draft.

v4 is the minimum-viable Catapult: the smallest spec that still
expresses the load-bearing ideas (reactive schema, meaning-engine
chain, phased delivery, event-sourced state) while cutting the
enterprise surface area v3 carried — multi-user write
collaboration, deep governance, AI sandboxing, queueing flow
lobby, server-driven orchestration. The cut is deliberate:
v3's superset is the right product over time; v4 is what gets
built first so a single user driving via Claude Code can
actually use it.

The spec is structured as three parts. **Part A** describes the
platform — the abstract machinery that every Catapult deployment
runs regardless of bundle. **Part B** describes the default
bundle — the specific tier graph, prompts, grammars, and flow
definitions that ship as `bundles/default/` in a fresh project.
**Part C** describes the implementation architecture —
Elixir/OTP, Postgres, Commanded, Phoenix/LiveView, the remote
MCP server, the local Go CLI.

The line between Part A and Part B is the configurability lever.
A different bundle could replace every concrete decision in
Part B (which tiers exist, what they produce, how they chain)
without touching anything in Part A. The default bundle is one
worked example of the platform contract.

# Part A — Platform

## A.0 Vision

Catapult is a **design memory** system. Software design happens
once, in scattered conversations across architecture documents,
RFCs, code review threads, and tribal knowledge that leaves with
the people who hold it. Catapult is the machine that catches
that work as it happens, structures it into a traversable graph
of artifacts, and keeps the *why* attached to every decision so
the system as a whole remains explicable to humans and to
agents long after the original author has moved on.

The artifacts Catapult produces are not just documentation. They
are the structured input to AI code generation. A component
architecture document with a specific public API surface, a
specific set of dependencies, a specific policy section, and a
specific failure-handling discipline is what an AI agent needs
in order to write that component's code without inventing the
contract. The closer the artifacts hew to a stable, queryable,
machine-readable structure, the more useful they are downstream
— and the more reliably they capture the design intent the
author would otherwise leave implicit.

The chain of artifacts is generated by an LLM driven by a human
user. The user works through Claude Code; Catapult is the
substrate Claude Code reads, writes, and reasons about. There
is no separate Catapult UI for *generation* — there is a
dashboard for *observation* (the structured graph, the review
queue, the phase plan), but the working interface is a CC
session in front of a project repository.

### A.0.1 The load-bearing architectural commitments

Four commitments shape every later chapter and explain what
Catapult does and doesn't do:

1. **Claude Code is the driver.** The user opens a CC session
   and works through skills + slash commands that read state,
   compose artifacts, commit them, and advance the chain. There
   is no server-side workflow engine that "runs" a project.
   Catapult does not call LLMs; CC does.
2. **The server is pure state.** Catapult's job is to hold the
   event log, derive projections, enforce invariants at command
   time, and serve "what's ready next" to the agent. It reacts
   to commits the agent makes. It does not initiate work, does
   not poll for things to do, does not run background
   generation, does not orchestrate flows. Workflow lives in
   the agent's skill suite, not in the server.
3. **State lives in Postgres; artifacts live in git.** The body
   files an LLM produces (the markdown+XML designed content)
   live in the user's project git repo and are pushed to a git
   remote Catapult can fetch from. Everything else — status,
   draft history, reviews, approvals, edges, propagation
   records, the phase plan, the event log itself — lives in
   Catapult's Postgres. The split is clean because bodies are
   the only thing humans want to `git diff` across iterations,
   and state is the only thing the system needs to query.
4. **Single user has write access; collaborators have read.**
   v4 deliberately does not support multi-user concurrent
   write. The project owner drives. Collaborators can log in to
   the dashboard, see the structured graph, read reviews, and
   pull the project repo locally — but they cannot drive the
   chain. Multi-user write would require either commit-ordering
   coordination or aggregate-level merge logic; v4 sidesteps
   both by making write a single-driver concern.

These commitments rule out a substantial chunk of what an
enterprise-grade design memory would do, and that's the point.
v4 is the smallest version that can demonstrate the core idea
end-to-end. The features cut from v3 — bundle-config-management
UI, multi-reviewer assignment with scoped roles, AI sandboxing,
deep audit governance, real-time external integrations, complex
credentials with token-quota tracking — are deferred to a later
v5 that builds on a validated v4 foundation rather than ahead of
it.

### A.0.2 The self-bootstrapping invariant

Catapult is built by reading this specification through
**SiegeEngine**, the predecessor system. SiegeEngine's substrate
walks an input document through a tier chain, drafts and reviews
each artifact, and commits the resulting bodies to a git
repository. Pointed at this spec, SiegeEngine produces the body
files for Catapult's own architecture — feature expansions,
requirements, sysarch components, comparch decompositions, all
the way down to implementation artifacts a coding agent reads
to write the code.

The artifacts SiegeEngine produces in that bootstrap pass must
be **byte-identical** to what Catapult, once running, would
produce for the same input. The reason this matters: it's the
test that the specification is honest. If Catapult's spec says
"the chain produces these artifacts via this prompt sequence"
and SiegeEngine produces something different running the same
prompts on the same input, the spec is wrong somewhere — the
prompts diverge, the readiness rules diverge, the grammar
diverges. The byte-identity check forces the spec to be
self-contained: it cannot reference SiegeEngine's internals to
explain Catapult's behavior, because SiegeEngine doesn't know
about itself when it's reading the spec as an input document.

Once SiegeEngine has produced Catapult's body files, a fresh
Catapult instance ingests them: fetches the project branch,
walks the body tree, materializes the equivalent state in
Postgres, and is then the running system. SiegeEngine has done
its job. Subsequent design iteration happens through Catapult.

## A.1 The structured model

Catapult organizes design work into **tiers**, each of which
holds a set of **scopes**, each of which has a **body** (and
optionally a **review**) stored in git. Tiers are arranged in a
chain by dependency edges; the chain is what makes a design
project a connected graph rather than a pile of documents.

This chapter describes those concepts as the platform exposes
them. The specific tiers a project uses, the shape each tier's
scopes take, the grammar each tier's body follows — all of that
is bundle content (§A.2). The platform contract is what's
stable across bundles.

### A.1.1 Tiers

A **tier** is a layer of structured artifacts produced at one
level of abstraction. Each tier in a chain transforms its
inputs into outputs that the next tier consumes. The
transformation differs per tier: a tier may *extract* meaning
from raw input prose, *rotate* an axis of decomposition (e.g.
turn user-facing features into system-level responsibilities),
*compress* a many-element layer into a smaller one (e.g. group
responsibilities into components), or *expand* a single element
into its constituent leaves (e.g. carve a component into
subcomponents). The chain alternates these transformations as
the project descends from problem statement to implementable
leaves.

A tier is identified by a string name (e.g. `feature_expansion`,
`comparch`, `impl`). The string is opaque to the platform but
load-bearing in the bundle, which uses it as the key under
which prompts, grammars, scope shapes, and readiness predicates
are registered.

Tiers chain via **dependency edges**. A tier's readiness
predicate (§A.2) names the upstream tiers whose state it reads;
those reads are the chain's structural arrows. The chain
graph the dependency edges form must be acyclic — enforced at
bundle-load time by the platform (§A.2.4) — because cycles
would prevent the readiness queries from ever terminating in a
"this scope is ready to draft" answer.

A tier may also declare a **review** pass that critiques its own
output (§A.6). Reviews are advisory: an approved body is
considered settled regardless of whether the review has run or
what it says. The review's value is iteration signal, not gate.

### A.1.2 Scopes

A **scope** is a stable key identifying one artifact within a
tier. Scopes are *the* unit of work the platform reasons about:
"the comparch for component `comp_a`" is one scope; "the impl
for subcomponent `sub_b` of `comp_a` in phase 2" is another.
Every body file in git corresponds to one scope; every event
in the log targets one scope; every readiness query returns
the scopes of one tier that are ready to draft.

A scope's key is a tuple of dimensions. Which dimensions a
scope carries depends on its tier. The platform recognizes
four dimensions that any tier may use:

- **`comp_id`** — names a top-level entity (a feature, a
  responsibility, a component, or a project-level singleton
  using the sentinel value `proj`). Most tiers use this as
  their primary dimension.
- **`parent_id`** + **`sub_id`** — names a sub-element within
  a parent. Sub-tiers (subcomparch, sub-impl) use the pair to
  identify "the sub-thing inside this parent." `parent_id`
  alone identifies the parent; `sub_id` alone is meaningless
  without it.
- **`phase`** — partitions a tier across delivery phases (§A.7).
  Set only on tiers that the bundle declares as phased.

Every scope has a deterministic path in git derived from its
key: `<tier>/<scope_path>/body.md` for the body,
`<tier>/<scope_path>/review.md` for the review (when present).
The path layout for each tier is bundle content; the platform
guarantees that the layout is a pure function of the key, so
two scopes with the same key always resolve to the same path.

Scope keys are minted by the bundle's tier-specific
**identity rules** at the point where a parent tier's body
declares a downstream scope. For example, a sysarch body
declaring `<component alias="auth">` triggers Catapult to mint
a `comp_*` id and bind it to that alias. The minted id is
recorded as Postgres state, not as a file in git — there are
no identity ledgers on disk in v4 (§A.10).

### A.1.3 Phases

A **phase** is an optional dimension on a scope. A bundle
declares which tiers carry phase as a key dimension; tiers
that don't are *unphased*, meaning one body per scope-key for
the whole project. Tiers that are phased get one body per
`(scope_key, phase)` pair: a leaf subcomponent gets one impl
body for each phase it participates in, and a domain
component's fanin synthesis recomputes per phase.

Phases exist because delivery is incremental. A project with
twenty subcomponents doesn't get every impl written at once;
the user defines a phase plan that says "in phase 1 we build
auth + the foundation; in phase 2 we add billing on top." Each
phase's impl bodies see the full design intent for their
subcomponent but only the impls + fanins of the prior phases
as their built reality. Cross-phase delta context — "what
changed between phase 2's plan and phase 3's plan that this
re-implementation needs to know" — is a bundle-declared input
to a phased tier's prompt (§A.7).

The phase dimension lives on scopes, but the phase **plan** —
which phases exist, which scopes participate in each, the
build order within a phase — lives as a Postgres projection,
not as a per-scope artifact. The plan is computed from the
project's structural state (the comparch decompositions, the
dependency edges, the user's phase assignments) and re-derived
whenever that state changes. §A.7 covers the plan's full
shape and how the agent walks it.

### A.1.4 Bodies

A **body** is the per-scope artifact in git: a markdown file
with embedded XML sections that carry the structured content
downstream tiers parse. The body is the actual designed
content — the prose the LLM produced — and the only thing
humans want to `git diff` across iterations.

The body's grammar is per-tier and bundle-declared (§A.2).
Different tiers' bodies look different: a feature expansion
body is mostly `<feature>` blocks with intent prose; a
comparch body has typed sections for techspec, public API,
private API, policies, failure surface, and subcomponent
decomposition. The grammar is enforced at commit time by the
platform's body validator (§A.4), which runs the bundle's
per-tier grammar against the submitted body and rejects
commits that don't parse.

A body's content is the source of truth for the structured
fields the platform projects out of it. The Postgres
projections (nodes, edges, fragments, dependency graphs,
component decompositions) are *derived* from body content;
they are never the canonical store. Re-projecting a clean
event log + the bodies at a given git sha must produce
byte-identical projection state every time, regardless of what
order events arrived in or whether the projection was rebuilt
incrementally or from zero.

### A.1.5 Reviews

A **review** is the AI-produced critique of a body. Reviews
live in git next to the bodies they review:
`<tier>/<scope_path>/review.md`. Putting reviews in git makes
them diffable across iterations — a reviewer can compare the
previous review's findings against the current one and see
which issues the regeneration addressed.

Review content is structured: an intro paragraph (the "how
close to finished" read), a numeric score (0-100), and two
sections of individually-addressable findings — handles +
structure findings and architectural-decision findings. The
review's grammar, like the body's grammar, is bundle-declared.

Reviews are advisory. Approving a body does not gate on the
review completing or passing. The user can read the review and
decide whether to feed any of its findings back into a
regeneration; approval and review are independent lifecycles
that happen to share a draft.

### A.1.6 The chain as a graph

The tier set, with dependency edges between tiers, is a
directed acyclic graph. The chain is its topological
realization — the order in which tiers can be drafted given
that each tier's readiness requires its dependencies' content.

Most bundles will produce a tree-shaped chain (one tier feeds
several downstream tiers; convergent points where multiple
upstreams feed one downstream tier are rare but legal). The
platform makes no assumption about shape beyond acyclicity:
the bundle declares the edges, the platform validates the
graph, the reactive engine (§A.3) walks the projected state
to decide which scopes are ready at any moment.

Cross-tier edges aren't only dependencies. The bundle may also
declare *decomposition* edges (e.g. a sysarch component
decomposes into subcomponents) and *reference* edges (e.g. a
comparch may cite a project-level reference document). The
platform tracks edge types as part of the projection so the
dashboard graph view can render them with distinct visual
semantics, but only dependency edges feed readiness.
